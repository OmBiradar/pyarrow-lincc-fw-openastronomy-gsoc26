Today I got to meet with the arrow developers along with @hombit

They provided various thoughts to go through before finalizing a good PR for arrow.

There were the below points (notes by @hombit):

* Check with different compiler optimization flags, for the PR we will need different CPUs (with Github actions we can try ubuntu-24.04-arm and macos-latest runners, we can also connect a self-hosted larger Intel CPU runner)
* Compare performance to different parquet readers: datafusion, duckdb, polars
* PR would require the use of the arrow C++ internal benchmark system, but it is not in the great state, ask for help if needed <https://arrow.apache.org/docs/developers/benchmarks.html>
* Join the Zullip chat
* Interesting information:
* Proposal for parquet-native support of fixed-length arrays <https://docs.google.com/document/d/1nf30OqK_UqxA4YTEZQszmOBEG56m9M5mp9rIYC2SUWc/edit?tab=t.0>
* run-end-encoded array in arrow <https://arrow.apache.org/docs/format/Columnar.html#run-end-encoded-layout>
* Should we try to optimize parquet write defaults for pyarrow?
* why not zarr.dev?
* Parquet Variant <https://arrow.apache.org/docs/format/CanonicalExtensions.html#parquet-variant> <https://clickhouse.com/blog/a-new-powerful-json-data-type-for-clickhouse>
